## 1. 引言（Introduction）

### 1.1 编写目的

本文档用于对 Smart Teaching Service System（STSS）的业务范围、功能需求、非功能需求、验证标准以及六个子系统之间的协作关系进行统一描述。该文档是课程评审与后续子系统 1.0 版本验收的重要依据。通过统一的章节结构、术语口径、编号体系和图示要求，本文档保证需求描述、后续设计与最终验证之间可以建立清晰的追踪关系。

### 1.2 项目范围

STSS 是一个面向高校教学场景的智慧教学服务平台，覆盖基础信息管理、自动排课、智能选课、论坛交流、在线测试和成绩管理六个子系统。本文档描述这些子系统在业务层面需要支持的用户角色、核心功能、状态变化、数据流转关系、质量约束和验证标准。

本文档覆盖内容包括：

- 六个子系统的用户场景、功能需求与非功能需求。
- 系统级上下文、角色边界和跨子系统业务流程。
- 课程要求中明确提出的 Data Flow Diagram、State Diagrams、Class Diagrams、CRC Cards 和 Validation Criteria。

本文档不直接展开以下内容：

- 数据库表字段、索引和 ORM 映射等实现级设计细节。
- REST API 路径、请求响应字段等接口级实现细节。
- 具体前后端页面布局、组件拆分和代码实现方案。

## 2. 系统总体描述（Overall Description）

### 2.1 产品定位与建设目标

STSS 的目标是构建一个面向高校教学全过程的综合性服务平台，使学生、教师、教务管理人员和系统管理员能够在统一系统中完成基础信息维护、教学资源管理、课程安排、选课决策、课程交流、在线测试与成绩分析等关键活动。与单一工具型系统不同，STSS 强调六个子系统之间的数据联动和业务连续性：基础信息管理提供统一身份与课程基础数据，自动排课决定课程安排，智能选课为学生提供选课支持，论坛交流增强课程互动，在线测试支持过程性评价，成绩管理完成正式成绩记录和分析。

### 2.2 系统边界与上下文

STSS 的系统边界围绕“教学服务”展开，主要服务于校内教学管理与学习活动。系统内部包含六个业务子系统及其共享基础能力，系统外部则是参与系统交互的用户角色、可能依赖的邮件或通知能力，以及课程要求中提到的 AI 能力。

| 系统内部范围                       | 说明                                |
| ---------------------------------- | ----------------------------------- |
| 统一身份、用户、权限与基础课程信息 | 由 A 模块提供，是其他模块的基础支撑 |
| 教室资源、排课和课表输出           | 由 B 模块负责                       |
| 培养方案、选课、退选与 AI 辅助     | 由 C 模块负责                       |
| 公告、发帖、回复、检索与统计       | 由 D 模块负责                       |
| 题库、试卷、作答、评分与统计       | 由 E 模块负责                       |
| 成绩录入、修改、查询与分析         | 由 F 模块负责                       |

| 外部参与者/依赖 | 说明                                                     |
| --------------- | -------------------------------------------------------- |
| 学生            | 使用系统完成选课、交流、测试、成绩查看等活动             |
| 教师            | 使用系统完成课表查看、公告发布、题库维护、成绩录入等活动 |
| 教务管理人员    | 负责资源维护、排课、选课管理和部分跨系统协调             |
| 系统管理员      | 负责系统基础配置、权限管理和安全保障                     |
| AI 辅助能力     | 为 C 模块提供选课建议与解释支持                          |

### 2.3 用户角色与干系人

| 角色                    | 核心职责                                                                         | 关注的主要子系统 |
| ----------------------- | -------------------------------------------------------------------------------- | ---------------- |
| 学生                    | 维护个人信息、制定培养方案、选课退课、参与课程交流、参加测试、查询成绩           | A、C、D、E、F    |
| 教师                    | 维护个人信息、查看课表、发布公告、发帖交流、维护题库、组卷、录入和分析成绩       | A、B、D、E、F    |
| 教务管理人员            | 维护教师和学生基础信息、课程与教室信息、执行排课、控制选课窗口、处理特殊教务事务 | A、B、C、F       |
| 系统管理员              | 管理全局权限与安全策略，保障用户身份、日志与系统运行安全                         | A                |
| 课程教师/TA（验收视角） | 关注系统是否满足教学过程组织和课程考核要求                                       | B、D、E、F       |

在本文档中，若未特别说明，“管理人员”默认指具备相应业务权限的教务管理人员或系统管理员；涉及安全审计时，以系统管理员职责为准。

## 3. 系统级需求概览

### 3.1 系统目标与核心业务能力

从系统层面看，STSS 至少应具备以下核心能力：

| 编号        | 系统级功能能力                                                                       |
| ----------- | ------------------------------------------------------------------------------------ |
| `FR-SYS-01` | 系统应提供统一的用户、角色、权限和基础课程信息管理能力，为各子系统提供共享基础数据。 |
| `FR-SYS-02` | 系统应支持课程、教师、教室和时间资源的组织与排课处理，并能输出可查询的课表结果。     |
| `FR-SYS-03` | 系统应支持学生基于培养方案完成课程搜索、选课、退选与结果查询。                       |
| `FR-SYS-04` | 系统应支持围绕课程开展公告发布、发帖、回复、附件交流和内容检索。                     |
| `FR-SYS-05` | 系统应支持教师组织在线测试，学生完成作答，并生成自动评分与统计结果。                 |
| `FR-SYS-06` | 系统应支持教师和学生完成正式成绩录入、查询、修改控制和成绩分析。                     |
| `FR-SYS-07` | 系统应保证六个子系统在用户身份、课程对象、时间安排和数据追踪上的一致性。             |

### 3.2 跨子系统业务流程概览

STSS 的主要跨子系统业务链路如下：

1. A 模块首先维护用户身份、院系专业、课程基础信息和权限规则，为其他模块提供统一的业务基础。
2. B 模块基于课程、教师、教室和时间资源完成自动排课或手动调课，并形成可查询的课表。
3. C 模块读取培养方案、课程信息和开课安排，在规定时间窗口内为学生提供选课、退选和结果查询能力，并在需要时提供 AI 辅助建议。
4. D 模块围绕课程和开课实例支持公告发布、讨论交流和内容检索，增强教学过程中的互动。
5. E 模块围绕课程组织在线测试活动，包括题库管理、组卷、作答、自动评分与统计分析。
6. F 模块基于课程开设、选课结果和测试/考核相关数据，支持正式成绩录入、查询、分析与修改追踪。
7. 系统管理员和教务管理人员在全流程中承担跨模块维护、异常处理、权限控制和资源调度职责。

### 3.3 系统级用户场景总览（User Scenarios）

| 场景编号    | 场景名称                           | 参与角色           | 主成功结果                                                           | 前置条件                                           |
| ----------- | ---------------------------------- | ------------------ | -------------------------------------------------------------------- | -------------------------------------------------- |
| `US-SYS-01` | 学生完成从信息确认到选课的全过程   | 学生、教务管理人员 | 学生能够查看符合培养方案的课程并完成选课，系统记录选课结果与课表     | 学生账号、专业和培养方案信息已建立；选课窗口已开放 |
| `US-SYS-02` | 教师围绕课程开展教学互动与测试     | 教师、学生         | 教师能够发布公告、组织讨论、创建测试，学生能够参与交流并完成在线测试 | 课程已开设并存在对应课表                           |
| `US-SYS-03` | 教务管理人员完成资源维护与排课管理 | 教务管理人员、教师 | 管理人员能够生成排课结果、处理调课冲突并输出课表                     | 课程、教师和教室基础信息已存在                     |
| `US-SYS-04` | 教师录入成绩，学生查看结果与进展   | 教师、学生         | 教师完成成绩录入与必要修改，学生可以查询成绩、学分与统计结果         | 课程已结束或达到成绩录入条件；学生已有选课记录     |
| `US-SYS-05` | 系统管理员维护权限与审计安全       | 系统管理员         | 系统管理员能够维护权限、查看日志并保障系统访问安全                   | 系统已投入使用并存在多角色用户                     |

### 3.4 系统级数据流图（Data Flow Diagram）

下图给出 STSS 的系统级 Level 0 视图，用于说明主要外部角色与六个子系统之间的数据流关系：

```mermaid
flowchart LR
    student["学生"] --> A["A 基础信息管理"]
    student --> C["C 智能选课"]
    student --> D["D 论坛交流"]
    student --> E["E 在线测试"]
    student --> F["F 成绩管理"]

    teacher["教师"] --> A
    teacher --> B["B 自动排课"]
    teacher --> D
    teacher --> E
    teacher --> F

    admin["教务管理人员 / 系统管理员"] --> A
    admin --> B
    admin --> C
    admin --> F

    A --> B
    A --> C
    A --> D
    A --> E
    A --> F
    B --> C
    C --> D
    C --> F
    E --> F
    ai["AI 辅助能力"] --> C
```

说明：

- A 模块向其他业务模块提供统一用户、角色、权限和基础课程信息。
- B 模块产生的课表安排会影响 C 模块的选课组织以及教师、学生的课表查询。
- C 模块的选课结果会影响 D 模块课程参与范围、E 模块测试参与范围和 F 模块成绩对象范围。
- E 模块的测试结果可作为 F 模块成绩分析的辅助输入。

### 3.5 系统级非功能需求

| 编号         | 非功能需求         | 说明                                                                                           |
| ------------ | ------------------ | ---------------------------------------------------------------------------------------------- |
| `NFR-SYS-01` | 安全与权限控制     | 系统应根据用户角色对功能与数据访问进行控制，防止越权访问。                                     |
| `NFR-SYS-02` | 一致的用户体验     | 系统应保持统一的术语、角色命名、界面风格和业务对象表达方式。                                   |

### 3.6 总体验证思路与覆盖率目标（Validation Criteria）

系统级验证采用“需求编号 + 场景编号 + 验证编号”三位一体的追踪方式进行。每个关键功能需求都应至少对应一个用户场景和一个可执行或可检查的验证标准；第 10 章进一步提供功能域级三向追踪矩阵，用于审计主要需求、用户场景和验证标准之间的覆盖关系。每个子系统章节都必须显式包含用户场景、DFD、状态图、类图与 CRC Cards、非功能需求和验证标准，从而满足课程给出的最低内容要求。

本项目当前采用以下系统级验证标准：

| 编号        | 系统级验证标准                                                                                                                                                  |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `VC-SYS-01` | 文档必须显式包含 `Introduction`、`User Scenarios`、`Data Flow Diagram`、`State Diagrams`、`Class Diagrams`、`CRC Cards`、`Validation Criteria` 等课程要求内容。 |
| `VC-SYS-02` | A-F 六个子系统章节必须采用一致的子章节模板，以保证结构统一和便于追踪。                                                                                          |
| `VC-SYS-03` | 需求追踪矩阵中，主要功能需求中至少 90% 应建立到验证标准的明确映射。                                                                                             |
| `VC-SYS-04` | 全文术语、角色命名、图例和语言风格应保持一致，不得出现明显口径冲突。                                                                                            |

## 4. 子系统 A：基础信息管理需求

### 4.1 子系统目标与边界

基础信息管理子系统是 STSS 的基础支撑子系统，负责维护全系统共享的用户身份、角色权限、组织结构、课程基础数据和安全审计数据。其他子系统依赖 A 子系统提供的用户、教师、学生、院系、专业、课程和培养方案等主数据，完成排课、选课、论坛、在线测试和成绩管理业务。

#### 4.1.1 建设目标

| 编号        | 目标             | 说明                                                                              |
| ----------- | ---------------- | --------------------------------------------------------------------------------- |
| `GOAL-A-01` | 统一身份基础     | 为学生、教师、教务管理人员、系统管理员提供统一账号、认证状态和个人资料。          |
| `GOAL-A-02` | 统一权限基础     | 通过角色和权限控制不同用户在各子系统中的可访问资源和可执行操作。                  |
| `GOAL-A-03` | 统一组织基础     | 维护院系、专业、学生、教师和管理员之间的隶属关系。                                |
| `GOAL-A-04` | 统一课程基础     | 维护课程目录、课程分类、学分、学时、考核方式、先修课程和课程负责人等基础数据。    |
| `GOAL-A-05` | 统一培养方案基础 | 维护专业培养方案、总学分要求、必修/选修学分要求和课程组成，供 C、F 等子系统使用。 |
| `GOAL-A-06` | 安全与审计       | 提供登录保护、密码安全、令牌管理、关键操作日志和安全审查能力。                    |

#### 4.1.2 范围内能力

- 用户基本信息管理：新增、编辑、删除、查询、筛选学生和教师用户，并维护用户状态。
- 用户个人管理：用户维护个人联系方式、头像、密码等允许自行修改的信息。
- 认证管理：登录、登出、令牌刷新、注册、账号激活、忘记密码、重置密码、修改密码。
- 角色权限管理：维护角色、权限、用户角色关系，并基于角色控制访问边界。
- 院系与专业管理：维护院系、专业和其关联学生、教师、管理员、培养方案。
- 课程基本信息管理：维护课程代码、名称、学分、学时、类型、类别、院系、负责人、描述、考核方式、先修课程和状态。
- 培养方案基础管理：维护培养方案及其课程构成，为智能选课和成绩学分统计提供基础约束。
- 系统日志管理：记录登录、密码、用户、权限、组织、课程、培养方案等关键操作，并支持查询和审计。

#### 4.1.3 范围外能力

| 不属于 A 子系统的内容                  | 归属     | 边界说明                                                                 |
| -------------------------------------- | -------- | ------------------------------------------------------------------------ |
| 自动排课、教室分配、课表冲突处理       | B 子系统 | A 只提供教师、课程、课程容量等基础数据，不决定排课时间和教室。           |

| 课程论坛、公告、帖子、回复、附件交流   | D 子系统 | A 只提供用户身份和课程基础信息。                                         |
| 题库、试卷、在线作答、自动评分         | E 子系统 | A 只提供教师、学生和课程基础信息。                                       |
| 成绩录入、成绩修改审批、GPA 和成绩分析 | F 子系统 | A 只提供用户、课程、培养方案和权限基础。                                 |

### 4.2 用户角色与用户场景（User Scenarios）

#### 4.2.1 角色定义

| 角色         | 角色代码建议                      | 核心权限边界                                                                         |
| ------------ | --------------------------------- | ------------------------------------------------------------------------------------ |
| 学生         | `student`                         | 查看和维护本人资料，修改本人密码，查看本人权限和与本人相关的课程基础信息。           |
| 教师         | `teacher`                         | 查看和维护本人资料，查看课程基础信息，作为课程负责人参与课程、测试、成绩等后续业务。 |
| 教务管理人员 | `admin`                           | 管理学生、教师、院系、专业、课程和培养方案等教务基础信息。                           |
| 系统管理员   | `super_admin`                     | 具有最高权限，可管理用户、角色、权限、状态、删除操作和全局配置。                     |
| 安全管理员   | `security_admin` 或管理员安全职责 | 审查系统日志、登录异常、密码重置和权限变更记录，处理安全事件。                       |

#### 4.2.2 用户场景

| 编号      | 场景名称         | 主要参与者               | 前置条件                                             | 主成功场景                                                                                                 | 异常与约束                                                                   |
| --------- | ---------------- | ------------------------ | ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `US-A-01` | 创建学生账号     | 教务管理人员、系统管理员 | 教务管理人员已登录并具备创建用户权限；专业信息已存在 | 管理人员录入用户名、初始密码、姓名、邮箱、学号、专业、年级、班级并分配学生角色；系统创建账号并返回用户详情 | 用户名、邮箱、学号重复时拒绝创建；密码不符合策略时拒绝提交。                 |
| `US-A-02` | 创建教师账号     | 教务管理人员、系统管理员 | 院系信息已存在                                       | 管理人员录入教师工号、姓名、邮箱、所属院系、职称、办公地点并分配教师角色                                   | 工号或邮箱冲突时拒绝创建；普通教师不得创建其他教师账号。                     |
| `US-A-03` | 查询和筛选用户   | 教务管理人员、系统管理员 | 用户数据已存在                                       | 管理人员按关键字、角色、状态、分页条件查询用户列表，并查看用户详情                                         | 未认证用户不能查询；普通用户只能查看本人详情。                               |
| `US-A-04` | 修改用户状态     | 教务管理人员、系统管理员 | 目标用户存在                                         | 管理人员将用户状态设置为启用、未激活或封禁；系统立即影响登录和访问能力                                     | 管理人员不得绕过权限修改更高权限账号；状态变更应记录日志。                   |
| `US-A-05` | 批量维护用户     | 系统管理员               | 待导入用户数据已准备                                 | 系统管理员批量创建用户或批量修改用户状态/角色，系统返回成功和失败统计                                      | 单次批量规模应受限制；任何失败项应给出明确原因。                             |
| `US-A-06` | 用户维护个人资料 | 学生、教师、管理员       | 用户已登录                                           | 用户修改本人邮箱、手机号、头像、性别等允许个人维护的字段                                                   | 用户名、学号、工号、角色、状态、所属组织等敏感字段不得由本人随意修改。       |
| `US-A-07` | 用户修改密码     | 学生、教师、管理员       | 用户已登录且知道旧密码                               | 用户输入旧密码和新密码，系统校验旧密码和密码强度，成功后吊销旧刷新令牌                                     | 旧密码错误或新密码不符合策略时拒绝；新旧密码相同应拒绝。                     |
| `US-A-08` | 忘记密码并重置   | 任意用户                 | 用户邮箱已登记                                       | 用户提交邮箱，系统发送重置链接；用户用有效令牌设置新密码，系统使令牌和旧会话失效                           | 不暴露邮箱是否存在；过期、重复使用或伪造令牌应失效。                         |
| `US-A-09` | 分配或撤销角色   | 教务管理人员、系统管理员 | 目标用户和目标角色存在                               | 管理人员为用户分配角色或撤销角色，系统重新计算用户权限并使旧令牌失效                                       | 非超级管理员不得分配超级管理员角色；管理员不得移除自己的最后一个管理员角色。 |
| `US-A-10` | 维护院系和专业   | 教务管理人员、系统管理员 | 管理人员已登录                                       | 管理人员新增、修改、查询、删除院系和专业，并查看学生、教师、培养方案关联情况                               | 删除院系前应确认无教师和专业关联；删除专业前应确认无学生关联。               |
| `US-A-11` | 维护课程基础信息 | 教务管理人员、系统管理员 | 院系、教师和先修课程信息已存在或可为空               | 管理人员维护课程代码、名称、学分、学时、类别、考核方式、负责人和先修课程                                   | 课程代码必须唯一；被培养方案引用的课程不应直接删除。                         |
| `US-A-12` | 维护培养方案     | 教务管理人员、系统管理员 | 专业和课程基础信息已存在                             | 管理人员创建培养方案，设置总学分、必修学分、选修学分，并增删改方案课程                                     | 方案课程应引用有效课程；课程类别和建议学期应满足基本校验。                   |
| `US-A-13` | 审查系统日志     | 安全管理员、系统管理员   | 系统已产生操作日志                                   | 管理员按用户、操作、资源、时间范围查询日志，追踪登录、密码、权限、删除等关键事件                           | 普通用户不得查看系统日志；日志不得包含明文密码或明文令牌。                   |
| `US-A-14` | 登录保护         | 任意用户                 | 用户账号存在                                         | 用户输入用户名和密码登录，系统返回访问令牌、刷新令牌、角色和权限                                           | 连续登录失败达到阈值后短时锁定；未激活、封禁或禁用账号不得登录。             |

### 4.3 功能需求

#### 4.3.1 用户基本信息管理

| 编号      | 功能需求                                                                                                                                                   | 优先级 |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| `FR-A-01` | 系统应支持系统管理员或具备授权的教务管理人员创建学生、教师和管理员用户。                                                                                   | Must   |
| `FR-A-02` | 创建用户时，系统应至少维护用户名、密码哈希、真实姓名、邮箱、手机号、性别、头像、状态、角色和创建/更新时间。                                                | Must   |
| `FR-A-03` | 创建学生用户时，系统应维护学号、专业、年级、班级；创建教师用户时，系统应维护工号、院系、职称、办公地点；创建管理员用户时，系统应维护管理员类型和所属部门。 | Must   |
| `FR-A-04` | 用户名、邮箱、学号、工号等唯一标识发生冲突时，系统应拒绝保存并返回明确错误。                                                                               | Must   |
| `FR-A-05` | 系统应支持分页查询用户列表，并支持按关键字、状态、角色或用户类型进行筛选。                                                                                 | Must   |
| `FR-A-06` | 系统应支持查看用户详情，包括基础信息、角色、学生/教师/管理员扩展信息、最后登录时间和状态。                                                                 | Must   |
| `FR-A-07` | 系统应支持更新用户基础信息，但应按权限区分本人可改字段和管理员可改字段。                                                                                   | Must   |
| `FR-A-08` | 系统应支持删除用户，删除前应校验操作权限，并清理或失效该用户相关认证令牌。                                                                                 | Should |
| `FR-A-09` | 系统应支持将用户状态维护为启用、未激活、封禁或禁用等状态，并在认证时严格检查状态。                                                                         | Must   |
| `FR-A-10` | 系统应支持批量创建用户，并返回总数、成功数、失败数和失败原因。                                                                                             | Should |
| `FR-A-11` | 系统应支持批量修改用户状态或角色，单次批量操作应设置数量上限。                                                                                             | Should |
| `FR-A-12` | 用户创建、更新、删除、批量状态变更等关键操作应写入系统日志。                                                                                               | Must   |

#### 4.3.2 用户个人管理

| 编号      | 功能需求                                                                                       | 优先级 |
| --------- | ---------------------------------------------------------------------------------------------- | ------ |
| `FR-A-13` | 已登录用户应能查看本人资料、角色和权限。                                                       | Must   |
| `FR-A-14` | 已登录用户应能修改本人邮箱、手机号、头像、性别等个人资料字段。                                 | Must   |
| `FR-A-15` | 用户不得通过个人资料维护功能修改用户名、学号、工号、角色、状态、所属院系、所属专业等管理字段。 | Must   |
| `FR-A-16` | 用户应能通过旧密码修改本人密码；系统应校验旧密码、新密码强度和新旧密码差异。                   | Must   |
| `FR-A-17` | 用户修改密码成功后，系统应使该用户现有刷新令牌失效，要求重新登录。                             | Must   |
| `FR-A-18` | 用户应能发起忘记密码流程；系统应以不泄露账号存在性的方式返回处理结果。                         | Must   |
| `FR-A-19` | 重置密码令牌应具备有效期和一次性使用约束，成功重置后同一用户其他重置令牌和刷新令牌应失效。     | Must   |
| `FR-A-20` | 用户上传或维护头像时，系统应校验头像地址或文件引用格式，避免写入非法资源地址。                 | Should |

#### 4.3.3 课程基本信息管理

| 编号      | 功能需求                                                                                                                                             | 优先级 |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| `FR-A-21` | 系统应维护课程基本信息，包括课程代码、课程名称、学分、学时、课程容量或默认容量、课程类型、课程类别、所属院系、课程负责人、课程描述、考核方式和状态。 | Must   |
| `FR-A-22` | 课程类型应至少支持专业必修课、专业选修课、公共课三类，以支撑培养方案和选课约束。                                                                     | Must   |
| `FR-A-23` | 系统应支持分页查询课程，并可按关键字、院系、课程类型和状态筛选。                                                                                     | Must   |
| `FR-A-24` | 系统应支持查看课程详情，包括课程负责人、所属院系、先修课程和更新时间。                                                                               | Must   |
| `FR-A-25` | 系统应支持创建、更新、删除课程；删除课程前应校验课程未被培养方案或其他业务对象引用。                                                                 | Must   |
| `FR-A-26` | 系统应支持维护课程先修关系，且先修课程必须引用已存在课程。                                                                                           | Should |
| `FR-A-27` | 系统应支持批量创建课程，并对每门课程返回成功或失败结果。                                                                                             | Should |
| `FR-A-28` | 课程代码应全局唯一；学分应为非负且符合一位小数精度要求；学时和容量应为非负整数。                                                                     | Must   |
| `FR-A-29` | 课程基础信息应作为 B、C、D、E、F 子系统共享主数据，其他子系统不得维护互相冲突的课程副本。                                                            | Must   |

#### 4.3.4 用户权限管理

| 编号      | 功能需求                                                                                             | 优先级 |
| --------- | ---------------------------------------------------------------------------------------------------- | ------ |
| `FR-A-30` | 系统应采用基于角色的访问控制模型，通过用户、角色、权限、用户角色关系、角色权限关系共同决定访问能力。 | Must   |
| `FR-A-31` | 系统应至少支持学生、教师、教务管理员、超级管理员，并可扩展安全管理员角色。                           | Must   |
| `FR-A-32` | 权限代码应按资源和动作组织，例如 `user:read`、`user:create`、`department:update`、`course:delete`。  | Should |
| `FR-A-33` | 系统应支持查询角色列表和用户权限列表，用户可查看本人权限，管理员可查看授权范围内用户权限。           | Must   |
| `FR-A-34` | 系统应支持为用户分配角色和撤销角色，角色变更后应使目标用户旧令牌失效。                               | Must   |
| `FR-A-35` | 非超级管理员不得分配或撤销会导致越权的高级角色。                                                     | Must   |
| `FR-A-36` | 管理员不得撤销自己的最后一个管理员或超级管理员权限，以避免系统失去管理入口。                         | Must   |
| `FR-A-37` | 所有受保护接口必须先完成身份认证，再进行角色或权限判断；未认证返回认证错误，无权限返回权限错误。     | Must   |
| `FR-A-38` | 系统应在权限变更、角色变更和越权拒绝场景中保留可审计记录。                                           | Should |

#### 4.3.5 系统安全管理

| 编号      | 功能需求                                                                                             | 优先级 |
| --------- | ---------------------------------------------------------------------------------------------------- | ------ |
| `FR-A-39` | 系统应通过访问令牌和刷新令牌支持登录会话，访问令牌用于短期认证，刷新令牌用于续期。                   | Must   |
| `FR-A-40` | 刷新令牌应以哈希形式存储，刷新时应采用轮换机制，已使用、过期或被撤销的刷新令牌不得再次使用。         | Must   |
| `FR-A-41` | 系统应记录登录成功、登出、密码修改、密码重置、用户状态变更、角色变更、课程和培养方案变更等关键事件。 | Must   |
| `FR-A-42` | 密码应以安全哈希形式存储，不得保存或记录明文密码。                                                   | Must   |
| `FR-A-43` | 密码强度应至少要求 8 位，并包含大写字母、小写字母和数字。                                            | Must   |
| `FR-A-44` | 连续登录失败达到阈值时，系统应对用户名和来源地址组合进行短时锁定。                                   | Must   |
| `FR-A-45` | 系统应支持账号注册和账号激活流程；激活令牌应具备有效期和一次性使用约束。                             | Should |
| `FR-A-46` | 系统日志应支持按用户、操作类型、资源类型、时间范围分页查询。                                         | Must   |
| `FR-A-47` | 系统日志不得记录明文密码、明文令牌、完整敏感凭据等信息。                                             | Must   |
| `FR-A-48` | 封禁、禁用或未激活账号不得登录，也不得刷新访问令牌。                                                 | Must   |

#### 4.3.6 院系、专业与培养方案基础管理

| 编号      | 功能需求                                                                                                 | 优先级 |
| --------- | -------------------------------------------------------------------------------------------------------- | ------ |
| `FR-A-49` | 系统应支持维护院系信息，包括院系名称、院系代码和描述。                                                   | Must   |
| `FR-A-50` | 系统应支持查询院系列表和详情，详情应展示关联专业、教师和统计信息。                                       | Must   |
| `FR-A-51` | 删除院系前，系统应校验该院系下无专业、教师、管理员或其他关键关联。                                       | Must   |
| `FR-A-52` | 系统应支持维护专业信息，包括专业名称、专业代码、所属院系、学位类型和总学分要求。                         | Must   |
| `FR-A-53` | 系统应支持按院系和关键字查询专业，并展示学生数量和培养方案信息。                                         | Must   |

### 4.7 非功能需求与约束

| 编号       | 分类         | 非功能需求                                                                                                               |
| ---------- | ------------ | ------------------------------------------------------------------------------------------------------------------------ |
| `NFR-A-01` | 安全性       | 所有非公开接口必须要求认证；涉及用户、角色、课程、培养方案和日志的管理接口必须执行角色或权限校验。                       |
| `NFR-A-02` | 密码安全     | 密码不得明文存储或明文输出；密码策略至少要求 8 位且包含大小写字母和数字。                                                |
| `NFR-A-03` | 令牌安全     | 刷新令牌必须哈希存储，一次刷新后旧刷新令牌立即失效；密码修改、密码重置、角色变更后应吊销相关刷新令牌。                   |

## 5. 子系统 B：自动排课需求

### 5.1 子系统目标与边界

自动排课子系统是 STSS 中负责“教室资源维护 + 排课结果生成 + 课表查询输出”的执行层子系统。B 模块在课程开设、学期、教师和教室等基础数据已经存在的前提下，为教务管理人员提供排课创建、冲突校验、手动调课和课表发布能力，同时为教师和学生提供课表查询能力。

#### 5.1.1 建设目标

| 编号        | 目标             | 说明                                                                     |
| ----------- | ---------------- | ------------------------------------------------------------------------ |
| `GOAL-B-01` | 统一教室资源管理 | 为排课提供教室基础资料维护、状态管理、容量管理和设备描述能力。           |
| `GOAL-B-02` | 统一排课结果生成 | 基于课程开设、教室和时间资源生成可执行的排课结果。                       |
| `GOAL-B-03` | 统一冲突校验     | 在保存前完成教室状态、容量与时间重叠检测，避免无效排课写入。             |
| `GOAL-B-04` | 统一课表查询     | 为教务管理人员、教师和学生提供按学期、按教室、按课程开设查询课表的能力。 |
| `GOAL-B-05` | 支持人工调整     | 在自动排课基础上保留人工调课、撤销和重新校验能力，满足教务实际操作。     |
| `GOAL-B-06` | 结果可追踪       | 排课结果、冲突结果和调整记录应可回溯，便于后续审计与问题定位。           |

#### 5.1.2 范围内能力

- 教室信息管理：新增、编辑、查询、删除教室，维护教学楼、校区、房间号、容量、类型、设备和状态。
- 可用教室查询：根据指定时间段、容量、校区和教室类型筛选可排课教室，并排除冲突资源。
- 排课预校验：在写入前完成课程开设存在性、教室状态、容量和时间冲突检查。
- 排课结果管理：新增、更新、删除、查询排课记录，并保留必要的备注信息。
- 自动排课：根据业务约束批量生成排课建议或正式排课结果。
- 手动调课：支持对既有排课进行时间或教室调整，并重新执行冲突校验。
- 课表查询：支持按课程开设、按教室、按学期查询课表，并输出适合前端渲染的数据结构。

#### 5.1.3 范围外能力

| 不属于 B 子系统的内容                    | 归属                     | 边界说明                                                 |
| ---------------------------------------- | ------------------------ | -------------------------------------------------------- |
| 课程基础信息维护、课程代码与课程属性管理 | A 子系统                 | B 只读取课程开设与课程基础信息，不定义课程主数据。       |
| 学期、课程开设、教师编排基础数据维护     | A 子系统或共享基础数据层 | B 依赖这些基础对象进行排课，不负责其生命周期管理。       |
| 培养方案审批、课程可选范围维护           | C 子系统                 | B 可读取培养方案相关数据用于课表展示，但不负责选课规则。 |
| 选课确认、容量占用、退选处理             | C 子系统                 | B 只提供排课结果，不管理学生选课状态。                   |
| 公告、交流、帖子与评论                   | D 子系统                 | B 仅为课表展示提供时间基础，不处理交流业务。             |
| 成绩录入、成绩分析与 GPA 统计            | F 子系统                 | B 只提供课程安排与课表查询，不负责成绩结果。             |

| `US-B-08` | 处理自动排课失败   | 教务管理人员、系统管理员 | 自动排课任务已启动             | 系统返回失败原因、冲突清单和不可用资源，管理员可据此调整约束后重新执行                       | 若资源不足或约束过严，应明确标识无法排课的原因，而不是返回空结果。 |

```mermaid
stateDiagram-v2
    [*] --> Draft: 创建任务

    Validating --> ValidationFailed: 校验失败
```

状态约束：

- `Published` 表示课表已对教师或学生可见。

### 5.7 非功能需求与约束

| 编号       | 分类     | 非功能需求                                                                               |
| ---------- | -------- | ---------------------------------------------------------------------------------------- |
| `NFR-B-01` | 正确性   | 排课写入前必须先执行服务端冲突校验，不能仅依赖前端预校验结果。                           |
| `NFR-B-02` | 一致性   | 教室状态、排课结果和课表查询结果应保持一致，状态变更后相关查询应及时反映。               |
| `NFR-B-03` | 冲突检测 | 同一教室在相同周次和节次范围内不得出现重叠排课；冲突提示应明确具体原因。                 |
| `NFR-B-04` | 可用性   | 教室状态为 `maintenance` 或 `unavailable` 时，系统不得返回其作为可排课资源。             |
| `NFR-B-05` | 性能     | 教室列表、可用教室和课表查询应支持分页与筛选，常规查询应保持可接受响应时间。             |
| `NFR-B-06` | 可调整性 | 自动排课结果应允许人工调整、撤销和重新校验，且调整后不能破坏已保存数据一致性。           |
| `NFR-B-07` | 可读性   | 课表查询结果应按周次、星期和节次组织，便于前端直接渲染周视图。                           |
| `NFR-B-08` | 可追踪性 | 自动排课和调课操作应记录任务状态、发起人、调整原因和冲突明细。                           |
| `NFR-B-09` | 数据约束 | 同一校区和教学楼范围内的房间号应保持唯一，容量必须大于 0，时间范围必须合法。             |
| `NFR-B-10` | 权限控制 | 教务管理人员和排课管理员可维护排课结果；教师仅能查询本人相关数据，学生仅能查询可见课表。 |

### 5.8 验证标准（Validation Criteria）

| 编号      | 覆盖需求                                     | 验证标准                                                                                                   |
| --------- | -------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `VC-B-01` | `FR-B-01` 至 `FR-B-08`                       | 创建、更新、查询和删除教室时，系统能正确保存基础字段和设备信息；同校区同教学楼下房间号重复时返回冲突错误。 |
| `VC-B-02` | `FR-B-04`、`FR-B-05`、`FR-B-06`、`FR-B-07`   | 将教室状态置为维护中或不可用后，该教室不再出现在可用教室列表中；容量为 0 或枚举非法时返回校验错误。        |
| `VC-B-03` | `FR-B-09`、`FR-B-11`、`NFR-B-01`、`NFR-B-03` | 对已存在排课时间段再次提交校验时，系统返回明确冲突类型和冲突教室信息；无冲突时返回通过。                   |
| `VC-B-04` | `FR-B-10`、`FR-B-12`、`FR-B-15`              | 执行自动排课任务后，系统能生成排课结果并输出冲突清单；资源不足时应给出不可排原因。                         |
| `VC-B-05` | `FR-B-13`、`FR-B-14`、`FR-B-16`、`FR-B-17`   | 调整排课教室或时间后，系统重新校验并保存；若调整后产生冲突，应返回 409 且不写入结果。                      |
| `VC-B-06` | `FR-B-18`、`FR-B-19`、`NFR-B-08`             | 删除或调整排课记录后，系统保留必要的审计痕迹和调整原因，便于后续追踪。                                     |
| `VC-B-07` | `FR-B-21`、`FR-B-22`、`FR-B-23`、`FR-B-25`   | 按课程开设、按教室和按学期查询课表时，返回结果与筛选条件一致；学生只能看到授权范围内的课表。               |
| `VC-B-08` | `FR-B-24`、`FR-B-26`、`NFR-B-05`、`NFR-B-07` | 课表查询支持分页和筛选，导出或打印结果与页面查询结果一致，且结果结构可直接用于周视图渲染。                 |

## 7. 子系统 D：论坛交流需求

### 7.1 子系统目标与边界

论坛交流子系统用于支撑“课程内通知发布 + 课程讨论互动 + 内容检索与统计”闭环。其核心目标是为任课教师、学生和论坛管理人员提供统一的课程交流空间，沉淀问题答疑、学习经验和教学通知，提升教学过程中的沟通效率。

#### 7.1.1 建设目标

| 编号        | 目标               | 说明                                             |
| ----------- | ------------------ | ------------------------------------------------ |
| `GOAL-D-01` | 统一课程公告发布   | 任课教师可发布课程公告，课程相关用户可及时查看。 |
| `GOAL-D-02` | 统一课程讨论入口   | 教师与学生可围绕课程发帖、回帖与交流。           |
| `GOAL-D-03` | 统一内容组织与治理 | 支持帖子状态管理、置顶、隐藏和删除等治理动作。   |
| `GOAL-D-04` | 统一检索能力       | 支持对历史帖子进行关键字检索与筛选。             |
| `GOAL-D-05` | 统一统计能力       | 支持发文数量和周/月热帖统计，辅助运营分析。      |

#### 7.1.2 范围内能力

- 公告管理：发布、查看、编辑、撤回课程公告。
- 发帖管理：教师与学生发帖，支持帖子类型、置顶与状态控制。
- 评论互动：用户对帖子回复，也可对评论进行二级回复。
- 附件关联：发帖时支持上传附件并在详情中展示。
- 帖子检索：按标题/正文关键字、课程、作者和时间筛选。
- 数据统计：按人、按课程、按周期统计发文和热帖。

#### 7.1.3 范围外能力

| 不属于 D 子系统的内容              | 归属         | 边界说明                                           |
| ---------------------------------- | ------------ | -------------------------------------------------- |
| 用户注册、认证、角色权限主数据维护 | A 子系统     | D 仅消费账号与角色信息，不维护认证逻辑。           |
| 课程基础信息与开课计划维护         | A/B/C 子系统 | D 读取课程与开课实例，仅用于论坛归属与可见性判断。 |
| 选课业务流程与选课结果维护         | C 子系统     | D 不负责选课规则和选课数据写入。                   |
| 在线测试组织、自动评分与题库维护   | E 子系统     | D 只承载交流内容，不处理测试流程。                 |
| 正式成绩录入、分析与复核           | F 子系统     | D 不写入任何正式成绩数据。                         |

#### 7.1.4 与其他子系统的数据依赖

```mermaid
stateDiagram-v2
```

状态约束：

- `Hidden` 仅管理角色可见，不对普通用户展示。
- `Deleted` 为逻辑删除状态，不允许继续编辑或评论。
- `Archived` 为历史归档状态，仅供审计或历史查询。

## 8. 子系统 E：在线测试需求

### 8.1 子系统目标与边界

在线测试子系统是 STSS 中负责“教学评价与过程性考核”的执行层子系统。E 模块在课程开设、师生身份和选课名单已存在的前提下，为教师提供题库维护、试卷编排、测试发布与成绩统计能力，为学生提供在线答题、实时计时与结果查询能力。系统支持客观题自动评分与主观题留白批阅，并与成绩管理子系统保持数据联动。

#### 8.1.1 建设目标

| 编号        | 目标               | 说明                                                                 |
| ----------- | ------------------ | -------------------------------------------------------------------- |
| `GOAL-E-01` | 统一题库资源管理   | 支持多课程、多题型的试题结构化录入、分类、检索与维护。               |
| `GOAL-E-02` | 统一试卷生成能力   | 支持教师手工组卷与基于约束条件的系统自动组卷。                       |
| `GOAL-E-03` | 统一在线作答体验   | 提供计时、防意外丢失、分段保存与正式提交的答题环境。                 |
| `GOAL-E-04` | 统一自动评分与统计 | 客观题提交后即时评分，并提供试卷维度、题目维度与个人维度的统计分析。 |
| `GOAL-E-05` | 支持测试过程治理   | 控制测试可见范围、开放与截止时间，保留异常中断与补考处理空间。       |
| `GOAL-E-06` | 结果可追踪与可同步 | 测试成绩应可追溯作答记录，并可作为过程性成绩同步至 F 子系统。        |

#### 8.1.2 范围内能力

- 题库管理：新增、编辑、删除、查询试题；支持单选题、多选题、判断题等题型；支持知识点标签与难度分级。
- 试卷管理：创建试卷、维护试卷结构、手工添加题目、设置分值与顺序。
- 自动组卷：按课程、题型、知识点、难度、数量等约束条件批量抽题生成试卷。
- 测试发布：将试卷与课程开设绑定，设置考试时间窗口、时长限制、重考策略与可见范围。
- 学生答题：在线浏览试卷、实时倒计时、自动暂存、异常恢复、正式提交与强制收卷。
- 自动评分：客观题提交后系统即时比对答案并计算得分；主观题预留教师人工批阅接口（本阶段可仅记录作答文本）。
- 统计分析：输出试卷平均分、最高/低分、得分分布、题目正确率、个人历史成绩与薄弱知识点分析。

#### 8.1.3 范围外能力

| 不属于 E 子系统的内容                    | 归属     | 边界说明                                                             |
| ---------------------------------------- | -------- | -------------------------------------------------------------------- |

### 8.5 状态图（State Diagrams）

#### 8.5.1 试卷与测试状态图

`FIG-8-03` 试卷/测试发布状态图。

```mermaid
stateDiagram-v2
    [*] --> Draft: 创建试卷
    Draft --> Published: 发布测试
    Draft --> Archived: 废弃
    Published --> InProgress: 到达开始时间
    Published --> Closed: 提前关闭
    InProgress --> Closed: 到达结束时间/手动结束

    Archived --> [*]
```

状态约束：

- `Draft` 仅创建者可见，可随意修改。
- `Published` 表示测试已生效，学生可在窗口期内进入，但试卷结构锁定。
- `InProgress` 表示已有学生正在作答或窗口已开启。
- `Closed` 表示答题窗口结束，不再接受新提交。
- `Archived` 仅用于历史查询与统计，不可恢复为可发布状态。

#### 8.5.2 学生答题会话状态图

`FIG-8-04` 答题会话状态图。

```mermaid
stateDiagram-v2
    [*] --> NotStarted: 未进入

    InProgress --> AutoSaving: 自动暂存/手动保存
```

- `TimeUp` 强制触发收卷逻辑，确保超时答卷不丢失。
- `Grading` 完成后，学生与教师可查看对应分数与明细。

## 9. 子系统 F：成绩管理需求

### 9.1 子系统目标与边界

成绩管理子系统（Subsystem F）的核心目标是对课程成绩进行统一、规范和可审计的管理，并在此基础上提供多维度的数据查询与统计分析能力。该子系统围绕 `CourseOffering` 这一课程开设实例，完成从成绩录入、草稿保存、正式提交，到提交后受控修改及最终分析展示的完整生命周期管理。

#### 9.1.1 子系统目标

1. 为任课教师提供结构化的成绩录入与提交流程，确保成绩录入过程规范、数据来源统一。
2. 为学生提供准确、及时的成绩查询与学业进展分析，包括 GPA、学分完成情况与培养方案进度。
3. 为教师与管理员提供课程维度的成绩统计分析能力，以支持教学质量评估。
4. 为所有成绩变更行为提供完整的审计追踪机制，确保成绩修改过程可追溯、可监管。

#### 9.1.2 范围内能力

- 课程成绩录入：支持教师按课程开设实例录入平时成绩、期中成绩和期末成绩。
- 草稿与提交管理：支持成绩草稿保存、反复修改和正式提交。
- 成绩修改控制：支持已提交成绩的修改申请、审批、驳回和日志追踪。
- 学生成绩查询：支持学生查看本人成绩、GPA、学分进展和培养方案完成情况。
- 成绩统计分析：支持教师和管理员查看课程平均分、分布、排名、通过情况等分析结果。
- 审计与权限隔离：支持成绩相关敏感操作日志记录，并按教师、学生、管理员角色限制访问边界。

#### 9.1.3 范围外能力

| 不属于 F 子系统的内容            | 归属     | 边界说明                                                                   |
| -------------------------------- | -------- | -------------------------------------------------------------------------- |

### 9.3 功能需求

#### 9.3.1 任课老师成绩录入管理

任课老师成绩录入管理功能用于支持教师围绕具体 CourseOffering 完成成绩录入、草稿保存、正式提交、提交后锁定以及修改申请发起等完整业务闭环。系统应允许教师按课程开设维度拉取学生名单与已有成绩记录，并以表格方式完成平时成绩、期中成绩和期末成绩的在线维护。对于尚未创建正式成绩记录的学生，系统应返回可录入的占位项并以 EMPTY 状态展示；教师在录入后可通过批量 Upsert 方式将成绩保存为草稿。成绩一旦正式提交，系统必须锁定普通编辑入口，教师若仍需调整成绩，应进入修改申请流程，由后续审批机制处理。前端可提供即时预览，但总评、绩点与等级等正式结果必须以后端计算并返回的值为准。

#### 9.3.2 学生成绩查询管理

| `FR-F-14` | 系统应支持学生按学期查询本人成绩记录，使学生能够查看指定学期内各开课实例对应的成绩情况。                                           | Must   |
| `FR-F-15` | 系统应支持学生按课程名称、课程代码或课程关键字筛选本人成绩记录，以便学生快速定位特定课程成绩。                                     | Must   |
| `FR-F-16` | 系统返回的学生成绩列表应至少包含课程名称、课程代码、学期、学分、总评成绩、绩点、成绩等级或通过状态等学校允许公开给学生的成绩信息。 | Must   |
| `FR-F-17` | 系统应仅向学生展示已经正式发布、确认完成或按学校规则允许学生查看的成绩记录；草稿、暂存、未提交或未公开成绩不得展示给学生。         | Must   |
| `FR-F-18` | 当学生暂无可查询成绩记录或筛选条件无匹配结果时，系统应返回可识别的空结果，前端应据此给出明确反馈。                                 | Must   |

#### 9.3.3 成绩分析功能

成绩分析功能用于支持教师和具备授权的管理人员从课程维度观察成绩总体质量、分布特征和学生表现差异。系统应围绕单个 CourseOffering 输出平均分、最高分、最低分、成绩分布、通过/未通过统计、提交进度以及排名等分析结果，并以图表或结构化数据方式呈现。该功能的分析结果应以后端统一口径计算为准，前端仅负责展示，不得将分析性统计逻辑分散实现于多个页面或组件中。

| 编号      | 功能需求                                                                                                                 | 优先级 |
| --------- | ------------------------------------------------------------------------------------------------------------------------ | ------ |
| `FR-F-19` | 系统应支持教师和具备授权的管理人员查看某个 CourseOffering 的课程成绩分析结果。                                           | Must   |
| `FR-F-20` | 课程成绩分析应至少包括课程平均分、最高分、最低分、已提交人数、通过人数和未通过人数等核心统计指标。                       | Must   |
| `FR-F-21` | 系统应提供课程成绩分布分析能力，并按统一分数段输出各区间人数或比例，以支撑柱状图、饼图或区间统计表展示。                 | Must   |
| `FR-F-22` | 系统应支持输出课程成绩排名结果，至少提供前若干名学生的学号、姓名、总评成绩和名次，用于展示课程 Top 排名。                | Must   |
| `FR-F-23` | 系统应提供课程维度分析能力，使教师能够从单门课程开设实例出发查看该课整体成绩质量，而不仅限于单个学生明细。               | Must   |
| `FR-F-24` | 系统应对课程分析结果采用后端统一口径计算，总评、绩点、等级、平均分、分布和排名等指标不得由前端作为业务真值自行计算。     | Must   |
| `FR-F-25` | 课程分析结果应支持前端以结构化方式展示，包括统计卡片、分布图、排名表或其他可视化组件，以增强教师对课程成绩结构的理解。   | Should |
| `FR-F-26` | 教师仅可查看本人任课课程的成绩分析；管理人员和超级管理员可在授权范围内查看更广泛的课程分析结果，但不得突破既定权限边界。 | Must   |

学生个人成绩分析功能用于支持学生了解本人学习成果和学业完成情况。系统应基于学生已正式发布的成绩记录、课程信息、学期信息和培养方案数据，计算并展示学生的 GPA、平均分、学分完成情况和培养方案进度。相关分析结果应与正式成绩记录保持一致，并随成绩确认、成绩修改或课程数据变化动态更新。学生只能查看本人分析结果，系统不得向学生返回其他学生的成绩统计信息。GPA、平均分和学分进展等核心指标应采用系统统一统计口径，避免不同页面或不同接口展示结果不一致。

| 编号      | 功能需求                                                                                                                                                                                                 | 优先级 |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| `FR-F-27` | 系统应支持学生查看本人 GPA 与平均分摘要，GPA 和平均分应由系统基于正式发布或确认可见的成绩记录动态计算并返回。                                                                                            | Must   |
| `FR-F-28` | 系统应支持学生查看本人学分完成情况，至少包括已修学分、通过学分、应修学分、已完成课程数和未通过课程数；其中应修学分应指当前学生所属培养方案规定的总应修学分或对应课程类别要求学分，而非当前已选课程学分。 | Must   |
| `FR-F-29` | 系统应支持学生查看本人培养方案完成进展，进展计算应基于培养方案、培养方案课程、课程和正式成绩记录之间的业务关联关系。                                                                                     | Should |
| `FR-F-30` | 若学生存在重修、补考或同一课程多次成绩记录，系统应按照学校成绩计算规则标识和处理相关记录；用于 GPA、平均分和学分完成情况计算时，应仅计入学校规则认定的有效成绩。                                         | Must   |
| `FR-F-31` | 系统应保证学生个人成绩分析仅基于当前登录学生本人的成绩、课程和培养方案数据生成，不得向学生返回其他学生的分析结果。                                                                                       | Must   |
| `FR-F-32` | 学生个人成绩分析结果应采用系统统一统计口径，避免 GPA、平均分、学分完成情况或培养方案进度在不同页面或不同接口中出现不一致。                                                                               | Must   |

### 9.5 状态图（State Diagrams）

成绩管理子系统涉及两个核心状态流：成绩记录状态流与成绩修改流程状态流。

#### 9.5.1 成绩记录状态图

成绩记录在生命周期中经历从未录入到最终确认的状态变化过程。

```mermaid
stateDiagram-v2
    [*] --> EMPTY
    EMPTY --> DRAFT : 教师录入
    DRAFT --> DRAFT : 修改/保存
    DRAFT --> SUBMITTED : 提交成绩
    SUBMITTED --> CONFIRMED : 系统确认/发布

    SUBMITTED --> MOD_REQUESTED : 发起修改申请
    CONFIRMED --> MOD_REQUESTED : 发起修改申请

    MOD_REQUESTED --> SUBMITTED : 审批通过或拒绝后回到提交状态
    MOD_REQUESTED --> CONFIRMED : 审批通过或拒绝后回到确认状态
```

#### 9.5.2 成绩修改流程状态图

该流程用于描述成绩在提交后发生修改时的受控流程。

```mermaid
stateDiagram-v2
    [*] --> NORMAL
    NORMAL --> REQUESTED : 教师发起修改申请
    REQUESTED --> APPROVED : 管理员批准
    REQUESTED --> REJECTED : 管理员驳回

    APPROVED --> UPDATED : 写入新成绩并记录日志
    UPDATED --> NORMAL : 关闭申请
    REJECTED --> NORMAL : 保持原成绩
```

上述状态流保证成绩一旦提交后不可直接修改，所有修改必须经过审批流程，并记录日志以供审计。

### 9.7 非功能需求与约束

| 编号       | 非功能需求                                                                                                                                                                  | 优先级 |
| ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| `NFR-F-01` | **数据准确性**：总评成绩、绩点和等级必须由后端统一计算，计算规则（平时30% + 期中20% + 期末50%）应集中维护，前端不得将自行计算的结果作为业务真值写入数据库。                 | Must   |
| `NFR-F-02` | **修改可追踪性**：所有正式成绩修改（审批通过后）必须同时写入 ScoreModificationLog 和 SystemLog，记录修改前后的分数差异、操作人和操作时间，以支持完整的审计追溯。            | Must   |
| `NFR-F-03` | **修改受控性**：成绩一旦提交（SUBMITTED），教师不得通过普通录入接口直接修改；所有提交后的修改必须经过"申请 → 审批"流程，系统必须在接口层拦截非法直接修改请求。              | Must   |
| `NFR-F-04` | **权限隔离**：教师只能操作自己任课的 CourseOffering；学生只能查询本人成绩；管理员可在授权范围内查看全部数据。任何越权请求必须在后端返回 403，不得依赖前端隐藏实现权限控制。 | Must   |
| `NFR-F-05` | **统计查询性能**：课程成绩分析接口（平均分、分布、排名等）在数据量不超过500人时，响应时间应控制在2秒以内，以保证教师查看分析结果时的基本可用性。                            | Should |
| `NFR-F-06` | **数据一致性**：草稿保存采用 Upsert 语义，对同一学生同一课程的重复保存不得产生重复记录，必须保证幂等性。                                                                    | Must   |
| `NFR-F-07` | **状态完整性**：成绩状态（EMPTY → DRAFT → SUBMITTED → CONFIRMED）的流转必须单向且受控，系统不允许状态倒退（如将 SUBMITTED 回退为 DRAFT）。                                  | Must   |
| `NFR-F-08` | **输入合法性**：所有成绩录入接口必须在后端使用 Zod 进行校验，分数范围限制为 0～100，非法输入必须返回明确的错误信息，不得静默写入。                                          | Must   |

### 9.8 验证标准（Validation Criteria）

本节定义成绩管理子系统的可验证标准（`VC-F-*`），用于指导测试与验收，确保系统功能覆盖率达到要求，并支持子系统版本 1.0 的验收测试。

| 编号      | 覆盖需求                                      | 验证标准                                                                                                                                                   |
| --------- | --------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `VC-F-01` | `FR-F-01` 至 `FR-F-05`、`NFR-F-06`            | Given 教师进入成绩录入页面，When 输入合法分数并保存，Then 系统成功保存为 `DRAFT` 且数据持久化；重复保存同一学生同一课程不得生成重复记录。                  |
| `VC-F-02` | `FR-F-07` 至 `FR-F-10`、`NFR-F-07`            | Given 成绩为 `DRAFT`，When 教师点击提交，Then 状态变为 `SUBMITTED`，后续普通编辑入口禁用，且提交前本地临时修改已同步至后端。                               |
| `VC-F-03` | `FR-F-06`、`NFR-F-08`                         | Given 输入非法分数（小于 0 或大于 100），When 保存或提交，Then 系统返回明确校验错误，非法数据不得写入正式成绩记录。                                        |
| `VC-F-04` | `FR-F-09`、`FR-F-11`、`NFR-F-03`              | Given 成绩已提交，When 教师尝试通过普通录入接口直接修改，Then 系统拒绝操作，并要求进入修改申请流程。                                                       |
| `VC-F-05` | `FR-F-11`、`FR-F-12`、`NFR-F-02`              | Given 教师提交修改申请，When 管理员批准，Then `Score` 更新，`modificationRequest` 清空，并生成 `ScoreModificationLog` 和 `SystemLog`。                     |
| `VC-F-06` | `FR-F-11`、`FR-F-12`、`NFR-F-03`              | Given 修改申请被拒绝，Then 原成绩保持不变，待处理申请标记消失，并记录驳回结果。                                                                            |
| `VC-F-07` | `FR-F-13` 至 `FR-F-16`、`FR-F-31`、`NFR-F-04` | Given 学生登录，When 查询成绩，Then 系统仅返回本人可见范围内的成绩数据，且包含课程、学期、学分、总评、绩点和等级等必要信息。                               |
| `VC-F-08` | `FR-F-17`                                     | Given 成绩未发布或仍处于草稿、暂存、未提交状态，When 学生查询成绩，Then 该成绩不可见。                                                                     |
| `VC-F-09` | `FR-F-18`                                     | Given 学生暂无成绩或筛选条件无匹配结果，When 查询成绩，Then 系统返回可识别的空列表或空结果提示，而不是错误响应。                                           |
| `VC-F-10` | `FR-F-19` 至 `FR-F-23`、`FR-F-25`、`FR-F-26`  | Given 课程存在成绩数据，When 教师或管理员请求课程分析，Then 系统返回平均分、最高分、最低分、通过人数、未通过人数、分布和排名等结果，并按权限限制可见范围。 |
| `VC-F-11` | `FR-F-24`、`NFR-F-01`                         | Given 不同页面或接口请求同一课程分析数据，Then 总评、绩点、等级、平均分、分布和排名等结果保持统一口径。                                                    |
| `VC-F-12` | `NFR-F-05`                                    | Given 课程人数不超过 500，When 请求课程成绩分析，Then 响应时间应控制在 2 秒以内。                                                                          |
| `VC-F-13` | `FR-F-27`、`FR-F-28`、`FR-F-29`               | Given 学生存在正式成绩和培养方案数据，When 查询 GPA 与学分进展，Then 系统返回 GPA、平均分、已修学分、通过学分、应修学分和培养方案完成进展。                |
| `VC-F-14` | `FR-F-30`                                     | Given 学生存在重修、补考或同一课程多次成绩记录，When 计算 GPA、平均分或学分进展，Then 系统仅按学校规则计入有效成绩。                                       |
| `VC-F-15` | `FR-F-32`、`NFR-F-01`                         | Given 多页面展示 GPA、平均分或学分进展，Then 不同页面和接口返回的数值一致。                                                                                |

### 10.3 关键业务流程覆盖说明

当前文档要求覆盖的关键业务流程包括但不限于以下几类：

1. 用户身份建立、登录与权限控制流程。
2. 教务资源维护、自动排课与手动调课流程。
3. 学生培养方案制定、选课、退选与结果确认流程。
4. 围绕课程的公告发布、发帖、回复与检索流程。
5. 教师组卷、学生在线答题、自动评分与统计分析流程。
6. 教师录入成绩、成绩修改控制、学生查询与绩点分析流程。

上述流程共同覆盖了 STSS 的主要业务主线，并在各子系统章节中对应到用户场景、功能需求和验证标准。
