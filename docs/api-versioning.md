# API 版本控制策略

## 当前版本

- **API 版本**: v1
- **基础路径**: `/api/v1`
- **发布日期**: 2026-03

## 版本策略

### 1. URL 路径版本控制

我们使用 URL 路径进行版本控制，格式为 `/api/v{version}`：

```
/api/v1/users     # 版本 1
/api/v2/users     # 版本 2（未来）
```

### 2. 版本升级流程

当需要发布新版本 API 时：

1. **创建新版本路由**
   - 在 `backend/src/routes/` 下创建 `v2/` 目录
   - 保留 `v1/` 目录不变

2. **并行运行期**
   - 新版本发布后，旧版本继续服务 **6 个月**
   - 在响应头中添加 `X-API-Deprecated: true` 标记废弃版本

3. **通知期**
   - 提前 **3 个月** 通知用户版本即将下线
   - 通过邮件、文档、响应头多种渠道通知

4. **下线**
   - 废弃版本返回 `410 Gone` 状态码
   - 响应体包含迁移指南

### 3. 向后兼容性承诺

#### 必须保持兼容的变更

- ✅ 添加新的可选字段
- ✅ 添加新的 API 端点
- ✅ 添加新的枚举值（在末尾）

#### 需要新版本的变更

- ❌ 删除或重命名字段
- ❌ 修改字段类型
- ❌ 修改必填/可选状态
- ❌ 修改错误响应格式

### 4. 版本生命周期

| 阶段           | 持续时间 | 说明               |
| -------------- | -------- | ------------------ |
| **Current**    | 活跃开发 | 当前推荐使用的版本 |
| **Supported**  | 12 个月  | 提供修复和支持     |
| **Deprecated** | 6 个月   | 仅维护，建议迁移   |
| **Retired**    | -        | 已下线，不可用     |

### 5. 版本响应头

每个 API 响应都包含以下头部：

```
X-API-Version: 1.0.0
X-API-Deprecated: false  # 如果版本废弃则为 true
X-API-Sunset: 2027-03-23  # 废弃版本的日落日期（ISO 8601）
```

### 6. 迁移指南示例

当 v2 发布时，在文档中提供：

````markdown
## 从 v1 迁移到 v2

### 变更 1：用户响应格式

**v1**: `{ id, name, email }`
**v2**: `{ id, name, email, createdAt }`

### 变更 2：分页参数

**v1**: `?page=1&limit=10`
**v2**: `?offset=0&limit=10`

### 代码示例

```typescript
// v1
const users = await fetch('/api/v1/users?page=1&limit=10')

// v2
const users = await fetch('/api/v2/users?offset=0&limit=10')
```
````

3. 订阅 API 变更通知

**服务端要求**：
1. 严格遵守语义化版本控制
2. 提供详细的迁移文档
3. 给予充足的迁移时间
