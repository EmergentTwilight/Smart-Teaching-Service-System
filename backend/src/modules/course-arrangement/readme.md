# 排课管理模块 (Course Arrangement) 后端开发说明

## 1. 模块概览与目录映射

本模块实现了排课系统的核心后端逻辑（接口 6.1 - 6.5）。

| **模块编号** | **接口功能名称** | **对应目录/文件** | **实现说明**                                           |
| ------------ | ---------------- | ----------------- | ------------------------------------------------------ |
| **6.1**      | **教室管理**     | `classroom/`      | 包含教室列表、查询、新增、修改、可用教室查询。         |
| **6.2**      | **排课记录管理** | `schedule/`       | 处理具体的排课记录 CRUD。                              |
| **6.3**      | **课表查询**     | `timetable/`      | 面向教师/学生的课表视图接口。                          |
| **6.4**      | **自动排课管理** | `auto-schedule/`  | **核心逻辑**。包含异步任务触发、进度查询、预览与应用。 |
| **6.5**      | **排课规则配置** | `rules/`          | 规则持久化存储。数据存放在 `data/rules.json`。         |

## 3. 重要注意事项（给后面同学的提醒）

已尽力保证实现与接口文档完全一致，但仍然由于一些原因（大组数据库设计、字段对齐、大组设计时没有rule表等问题）
在维护或优化代码时，请务必关注以下与文档不一致或潜在性能的地方：

### 自动排课算法 (6.4 - `auto-schedule.service.ts`)

1. **TeacherName 字段问题**：数据库 `Teacher` 表中并没有 `name` 字段（只有 `user_id` 等）。代码中已通过 `include` 关联 `User` 表，并使用 `user.name` 替代。如果前端强依赖 `Teacher` 直接返回 `name`，请注意此逻辑。
2. **暴力 DFS 算法**：目前的排课算法使用的是**暴力递归 (DFS)**，为了实现软约束偏好，遍历了所有可行课程组合去比较软约束评分，且没有剪枝。
   - 逻辑简单，能找到符合硬约束的解。
   - 若 `CourseOffering` 数量过大（如几百门课）且教室资源极度紧张，算法可能会陷入超长时间的搜索。后续同学可根据实际运行效果考虑引入超时强制截断或改用启发式算法。
3. **课表结构假设**：代码中暂时假设每节课为 **2 节连上**。如果实际业务中存在单节课或 3 节连上，需要修改 `periods` 枚举逻辑。

### 类型检查与数据库匹配 (6.1 - 6.3)

    **如果测试时发现问题，请首先关注这个问题，极有可能是大组数据库与B组接口文档或当前代码不一致导致的**，很抱歉当前实现可能没有验证完这类问题。

1. **Any 类型残留**：在早期开发的 `6.1 (Classroom)`、`6.2 (Schedule)`、`6.3 (Timetable)` 模块中，为了快速开发，部分地方使用了 `any` 类型。这可能导致与 `backend/prisma/schema.prisma` 定义的字段不完全匹配。
2. **类型优化**：`6.4` 和 `6.5` 已经实现了严格的 `types.ts` 定义。建议后续同学在测试 6.1-6.3 时，将 `any` 替换为正确的 Prisma 类型。

### 约束规则管理 (6.5)

    这部分由于大组数据库没有设计rule规则约束表，只能用json方式存储，如需优化且不嫌麻烦，可以跟大组长沟通修改数据库。

1. **仅支持增改，不支持删除**：目前的 `rule.service.ts` 严格按照文档实现，只提供了 `POST` 接口用于“设置（新增/更新）”。
2. **手动维护**：如果需要删除某条约束，目前只能手动编辑 `backend/src/modules/course-arrangement/data/rules.json`。后续如有业务需求，请自行添加 `DELETE` 接口。
